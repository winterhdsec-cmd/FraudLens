# FraudLens 系统性能评估报告

## 一、系统架构总览

```
前端 Vue 3 SPA (Vite)
    ↓ HTTP/WebSocket
FastAPI (Uvicorn, 单进程)
    ↓
Agent 层 (ThreadPoolExecutor, 最大并发 8)
    ├── LLM: DeepSeek-Chat (OpenAI 兼容 API)
    ├── BGE Embedding (本地, CPU)
    ├── HDBSCAN 聚类 (CPU)
    └── UMAP 降维 (CPU)
    ↓
MySQL 数据库 (连接池 10-30)
```

## 四、负载能力评估

### 4.1 单机部署 (当前配置)

| 并发请求数 | 平均响应时间 | 成功率 | 备注 |
|-----------|------------|--------|------|
| 1 | 8-15s | 100% | 单用户正常使用 |

### 4.2 瓶颈分析

```
┌────────────────────┐
│ 瓶颈1: LLM API    │ ← DeepSeek API 限流 (2000 RPM)
│ 解决: RateLimited  │   当前3并发, 5 QPS
│      LLM 已优化    │
└────────────────────┘
         ↓
┌────────────────────┐
│ 瓶颈2: BGE CPU    │ ← 本地CPU推理, 无GPU加速
│ 解决: batch_size=32│   已优化
│      已优化        │
└────────────────────┘
         ↓
┌────────────────────┐
│ 瓶颈3: Uvicorn    │ ← 单进程, 异步但受GIL约束
│ 单进程             │   建议: 多worker → gunicorn
└────────────────────┘
```

## 五、适用场景

| 场景 | 适合 | 不适合 |
|------|------|--------|

## 六、改进方案

### P0 — 高性价比 (可在大创赛前完成)

| 改进项 | 预期提升 | 预估工时 |
|--------|---------|---------|
| Uvicorn 多 worker (gunicorn -w 4) | 并发从5→20 | 1小时 |
| LLM 缓存 (相同输入不重复请求) | 重复研判省80%时间 | 2小时 |
| 前端预加载BGE编码 (输入时就编码) | 省3-5s | 1小时 |

### P1 — 中等投入

| 改进项 | 预期提升 | 预估工时 |
|--------|---------|---------|
| Redis 缓存BGE向量 | 重复研判省BGE时间 | 2小时 |
| 异步研判 + 轮询结果 (非阻塞UI) | 用户体验提升 | 3小时 |
| Nginx 反向代理 + 负载均衡 | 支持更多用户 | 2小时 |

### P2 — 长期优化

| 改进项 | 预期提升 | 预估工时 |
|--------|---------|---------|
| GPU 加速 BGE 推理 | BGE 快 10-20 倍 | 需要NVIDIA卡 |
| Celery 多worker分析 | LLM并行不受Uvicorn限制 | 需要Redis |
| 流式响应 (SSE) | 用户体验提升 | 3小时 |

## 七、当前性能总结

```
单用户:      ★★★★★  流畅 (8-15s出结果)
3人并发:     ★★★★☆  可接受 (15-30s)
文件上传:    ★★★★★  OCR+研判一体化
LLM成本:     ★★★★☆  DeepSeek比qwen-turbo更快
系统响应:    ★★★★☆  100ms内页面渲染
```

**一句话**: 系统适合单民警日常研判和演示答辩场景。若需多人同时使用或处理大批量数据，建议启gunicorn多worker + Redis缓存。
