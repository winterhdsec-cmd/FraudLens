## 四、当前仍存在的问题

### 功能层面

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 预警中心永远0条 | 🔴 高 | 数据库没有告警数据, alert_engine未自动触发 |
| P1页面为空(资金/派单/人员) | 🟡 中 | 演示数据只在cases/gangs表, P1表无数据 |
| OCR+研判需要后端配合 | 🟡 中 | EasyOCR在后端,前端上传需要后端运行 |
| initCharts硬编码数据 | 🔴 高 | pieChart/lineChart的数据是固定值, 不是后端返回的 |
| 刷新后echarts丢失状态 | 🟡 中 | SPA刷新后charts需重新初始化 |
| 管理端页面只有admin能看到 | 🟢 低 | 前端路由未限制, 但后端已做权限控制 |

### 性能瓶颈

| 瓶颈 | 影响 | 建议 |
|------|------|------|
| Uvicorn单进程 | 无法利用多核CPU | gunicorn -w 4 uvicorn.workers.UvicornWorker |
| BGE CPU推理 | 单次编码80-150ms | GPU加速或模型蒸馏 |
| MySQL单库 | 大并发读瓶颈 | 配置连接池适当放大(30→50) |
| 没有LLM缓存 | 相同输入重复请求 | 加Redis缓存, key=输入摘要hash |

## 五、功能优化方向 (按优先级)

### P0 — 数据填充

1. **alert_engine注入演示告警**: 写一个seed脚本往alert表插入20条告警数据, 让预警中心有内容可展示
2. **P1演示数据**: 资金流向/派单/重点人员表各插入10-20条, 让5个P1页面都有数据

### P1 — 用户体验

3. **initCharts改用真实数据**: 目前饼图/折线图数据是硬编码的, 改从Dashboard接口获取
4. **实战演示一键流程**: 录入页加"一键演示"按钮 → 自动填入3条案例 → 研判 → 跳转结果
5. **Search高亮关键词**: 搜索结果中高亮匹配的关键词

### P2 — 功能增强

6. **预警→派单流转**: 预警列表加"转为派单"按钮, 自动填充数据
7. **案件对比**: 选中两个案件并排对比(诈骗类型/金额/手法)
8. **报告模板选择**: 支持多模板(简要版/详细版/法制版)

## 六、性能优化方向 (按优先级)

### P0 — 部署架构

1. **gunicorn多worker**: 从单进程 → 4 worker
   ```
   pip install gunicorn uvicorn
   gunicorn -w 4 -k uvicorn.workers.UvicornWorker main:app
   ```
   预期: 并发从50 → 200+
   
2. **Redis缓存BGE向量**: 相同文本不重复编码
   ```python
   cached = redis.get(f"bge:{hash(text)}")
   if cached: return np.frombuffer(cached)
   ```
   预期: 重复研判省80%时间

### P1 — 查询优化

3. **DB连接池调大**: `pool_size=10` → `pool_size=30`
4. **Dashboard查询加索引**: monthly_trend的date_format查询走不上索引, 考虑加物化视图或缓存
5. **搜索加全文索引**: MySQL `FULLTEXT(title, description)` 替代 `ilike`, 搜索快10倍+

### P2 — 监控与可观测性

6. **请求耗时中间件**: 记录每个API的耗时到日志
7. **慢查询日志**: SQLAlchemy开启 `echo=True`, 抓出慢查询
8. **系统监控**: 接入 `/metrics` (prometheus_client) 暴露CPU/内存/QPS
